# Health: Agent 健康检查

## 职责

检查 agent 配置和 AI 编码环境是否健康。发现配置漂移、指令失效、验证缺失。

## 工作流

1. 逐层检查，收集发现
2. 按严重程度排序
3. 输出报告：
   ```
   ## Agent 健康报告
   
   ### 🔴 严重（影响功能）
   - <发现> → <修复建议>
   
   ### 🟡 警告（可能出问题）
   - <发现> → <修复建议>
   
   ### 🟢 正常
   - <检查项>: OK
   ```

## 检查项

### 1. 指令层

- AGENTS.md / CLAUDE.md / .cursorrules 是否存在？
- 内容是否过时（引用了不存在的文件/命令）？
- 多个指令文件是否有冲突？
- 指令中的路径是否可达？

### 2. 工具层

- 项目依赖是否安装完整？
- 测试/lint/typecheck 命令是否能正常运行？
- Git hooks 是否正常？
- MCP 服务器是否正常连接？

### 3. 验证层

- CI 是否通过？
- 本地验证命令（test/lint/typecheck）与 CI 是否一致？
- 有没有应该跑但没跑的验证？
- 验证命令是否指向正确的路径/配置（verifier 不要指向过时路径）

### 4. Skill 层

- 已安装的 skill 是否有冲突？
- skill 引用的路径/命令是否可达？
- 有没有过时的 skill？

### 5. Token 预算（budget-aware）

- 上下文窗口使用率
- 5 小时配额使用率
- 7 天配额使用率
- 哪些 skill/token 消耗最大

## 失败处理

| 触发条件 | 一线修复 | 仍失败兜底 |
|---------|---------|-----------|
| 配置文件缺失 | 检查默认位置和环境变量 | 标注"配置缺失"，给出创建命令 |
| 验证命令跑不起来 | 检查依赖安装和 PATH | 标注"验证未执行"，手动检查关键文件 |
| 指令冲突无法判断 | 列出所有冲突项让用户选择 | 以最新修改的指令为准 |
| 检查项太多导致报告太长 | 只输出 🔴 和 🟡 级别 | 按层级分批输出 |

## 🔴 CHECKPOINT · 修复确认

发现需要修复的问题时：
```
发现 <N> 个问题：
- [严重] <问题描述>
- [警告] <问题描述>

修复哪些？[全部 / 仅严重 / 跳过]
```

## 边界

- 只检查 agent 配置，不检查应用代码质量
- 一次性报告
- 不自动修复（除非用户明确要求）

## 反例

| ❌ 不要做 | ✅ 正确做法 |
|---|---|
| 自动修复发现的问题 | 报告后等用户决定 |
| 只检查配置文件不实际运行命令 | 实际跑一下验证命令 |
| 把应用 bug 当 agent 配置问题 | 区分 agent 配置 vs 应用代码 |
| 不检查就声称健康 | 逐项检查后报告 |
