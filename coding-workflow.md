# Coding Workflow

## 概述

将项目技能串联为端到端的开发工作流。收到任务请求后，从 `dispatch` 开始自动分流。

```
想法 → dispatch（自动分流）
        │
        ├── S级（零碎）→ 直接干
        │
        ├── M级（中等）
        │      ├── M-simple → implement（直接实现）
        │      └── M-phased → grill → dj-tdd → check
        │
        └── L级（完整）→ grill → output → dj-tdd → check → 完成
```

---

## Phase 0: 入口分流

### 触发条件

用户提出新任务、新需求、新想法时，自动进入 dispatch。

### 执行

```
/d:dispatch
```

dispatch 判断任务级别后，按以下路径继续：

| 级别 | 判断标准 | 路径 |
|------|---------|------|
| **S级** | 一句话能说清，<3 文件，无架构改动 | 直接干 |
| **M-simple** | 一次性改完，只涉及1层 | implement（不走 grill） |
| **M-phased** | 分步实现，涉及2+层 | grill → dj-tdd → check |
| **L级** | 新功能/架构改动，>10 文件，需求不确定 | grill → output → dj-tdd → check |

---

## Phase 1: S级路径 — 直接干

**适用**：改变量名、加日志、修 typo、改 1-2 个文件的简单逻辑。

**流程**：
1. 直接执行修改
2. 验证（typecheck / lint / 相关测试）
3. 完成

---

## Phase 2: M级路径

**适用**：3-10 文件的中等复杂度任务，需要确认方向但不需要完整设计。

### 2.1 M-simple（直接实现）

**判断**：改动只涉及1层，一次性改完就能交付。

**流程**：
```
直接实现 → 完成
```

- 不走 grill，不写 PRD，不走 TDD
- 告知用户"任务级别：M-simple，直接实现"，然后开干

### 2.2 M-phased（分阶段实现）

**判断**：涉及2+层，需要分步实现，每步需要验证。

**流程**：
```
grill（≤3问）→ 🔴确认 → dj-tdd → check → 完成
```

#### 2.2.1 grill（快速确认）

```
/d:grill
```

- 限制 3 个问题以内
- 每个问题附推荐答案
- 能从代码库找到答案的不问用户

**输出**：需求摘要（目标 + 范围 + 关键决策）

**CHECKPOINT**：需求确认后 → 进入 2.2.2

#### 2.2.2 dj-tdd（测试驱动实现）

```
/d:dj-tdd
```

- 红绿重构循环：RED → GREEN → REFACTOR
- 一次一个垂直切片
- 测试行为，不测实现

**完成后** → 结束

---

## Phase 3: L级路径 — 完整流程

**适用**：新功能、跨模块改动、需求不明确、需要调研和设计。

**流程**：

```
grill → 🔴确认1 → output → 🔴确认2 → dj-tdd → check → 完成
```

### 3.1 grill（深度对齐）

```
/d:grill
```

- 不限问题数，走完整决策树
- 用户给的 URL/参考材料全部读取
- 调研技术方案，搜索最新文档

**输出**：
```
## 需求摘要
- 目标：<一句话>
- 约束：<限制>
- 范围：<包含/不包含>
- 关键决策：<技术选型>
- 参考：<URL/文档>
```

**CHECKPOINT**：需求确认后 → 进入 3.2

### 3.2 output（创建文档）

```
/d:output
```

基于 grill 的输出，创建：
- `prd.md` — Problem Statement、Solution、User Stories、Implementation Decisions、Testing Decisions、Out of Scope
- 设计文档（复杂任务）— 背景、方案对比、选定方案详述、影响范围、验证方式

**CHECKPOINT**：文档确认后 → 进入 3.3

### 3.3 dj-tdd（测试驱动实现）

**L 级强制 TDD**：必须先写测试再写实现，红绿重构循环，不允许跳过。
```
/d:dj-tdd
```
- 红绿重构循环：RED → GREEN → REFACTOR
- 一次一个垂直切片
- 测试行为，不测实现
- 遵循深度模块原则
- 遵循阶梯决策（stdlib → 已有依赖 → 最少代码）

**完成后** → 进入 3.4

### 3.4 hunt ↔ check（审查循环）

代码实现后，进入 hunt 和 check 的循环直到质量达标。

**check（代码审查）**：
```
/d:check
```
- 功能完整性核对（对照 PRD 逐项确认）
- 代码质量审查（逻辑、错误处理、边界条件、命名）
- 过度工程检查（dj-ponytail 视角：delete/stdlib/native/yagni/shrink）
- 安全性检查

**发现问题时 → hunt（排查）**：
```
/d:hunt
```
- 构建反馈回路（失败测试 / curl / CLI）
- 最小化复现
- 假设 + 验证
- 修复 + 回归测试

**循环规则**：
- check 发现功能缺失 → 回到 3.3 dj-tdd
- check 发现 bug → hunt 定位根因 → 修复 → 回到 check
- check 通过 → 进入 3.5

### 3.5 output（文档对齐）

```
/d:output
```

- 检查代码实际行为与文档描述是否一致
- 代码改了 → 同步更新文档
- 功能偏移 → 标记并询问用户

**完成后** → 结束

---

## 按需路径

### 审计（独立执行）

需要全仓扫描时：
```
/d:audit
```

输出两份报告：
1. **过度工程报告** — 可删减项，按影响排序
2. **安全扫描报告** — 按严重程度（CRITICAL/HIGH/MEDIUM/LOW）

只报告，不修改。

### 调研（独立执行）

需要深度调研时：
```
/d:grill
```

- 模式 B：读 URL、参考文章、技术文档
- 输出调研摘要

## 失败回退

| 场景 | 处理 |
|------|------|
| dispatch 判断不了级别 | 偏保守选高一级（S→M, M→L） |
| grill 问了太多轮 | 压缩剩余问题，用推荐答案填充未决项 |
| 实现遇到 PRD 不清 | 回到 grill 补充对齐 |
| check 发现功能缺失 | 回到 dj-tdd 补充 |
| hunt 无法构建反馈回路 | 降级到手动复现步骤 |
| 文档和代码对不上 | 以代码为准，文档标注"待确认" |

## 改动回归检查

**原则**：改 A 不坏 B。每次改动一个 skill 或文档后，必须检查：

1. **引用一致性**：grep 搜索被改内容的旧措辞，确认所有引用点已更新
2. **关联文件**：README.md、coding-workflow.md、test-prompts.json 是否与新行为匹配
3. **跨 skill 引用**：其他 SKILL.md 是否引用了被改 skill 的旧行为
4. **流程完整性**：用测试 prompt 追踪执行路径，确认每条路径可达

发现不一致 → 立即修复，不要留到以后。

## 使用示例

### 示例 1：S级任务

```
用户：把 UserService 里的 log 改成 debug 级别

→ dispatch：S级，直接干
→ 直接修改，验证，完成
```

### 示例 2：M级任务（M-simple）

```
用户：给表单加个邮箱格式校验

→ dispatch：M-simple（只改接口层）
→ implement：实现校验逻辑
→ 完成
```

### 示例 3：M级任务（M-phased）

```
用户：重构用户模块，数据层+接口层都需要改

→ dispatch：M-phased（涉及2+层）
→ grill（≤3问题）：数据模型？接口设计？兼容性？
→ 🔴 确认
→ dj-tdd：测试驱动实现
→ check：审查代码+回归检查
→ 完成
```

### 示例 4：L级任务

```
用户：加个支付模块

→ dispatch：L级
→ grill：支付渠道？退款流程？对账需求？
→ 🔴 确认1
→ output：PRD + 设计文档
→ 🔴 确认2
→ dj-tdd：测试驱动实现
→ check：审查代码
→ output：文档对齐
→ 完成
```
