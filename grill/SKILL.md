# Grill: 深度对齐

## 职责

把模糊的想法变成清晰的、可执行的需求。包含三个子能力：
1. **对齐** — 通过提问消除歧义
2. **调研** — 读 URL、参考文章、技术文档
3. **建模** — 维护项目领域术语表（CONTEXT.md）

## Outcome Contract

- **产出**：模糊想法变成决策完整的推荐方案或执行计划
- **完成标准**：目标、成功标准、约束、选定方案、拒绝的备选、验证步骤都足够具体，可以直接执行而不需要重新决策
- **证据**：当前代码库状态、项目文档、外部参考、用户明确偏好
- **输出**：一个推荐方向或执行计划（含假设和验证步骤）

## 核心立场

**直接给意见。** 选一个立场，说明什么证据能改变它。避免：
- "这有很多思路"
- "你可以考虑"
- "各有优劣"

**不写代码直到用户批准。** 对齐完成前不写任何代码、脚手架、伪代码。

## 工作模式

### 模式选择路由

收到任务后，按以下规则选模式，不纠结：

| 信号 | 模式 | 判断依据 |
|------|------|---------|
| 用户给了 URL / PDF / 参考链接 | B：调研+对齐 | 有外部材料需要先读 |
| 用户说"看看这个库/框架/方案" | B：调研+对齐 | 需要先了解再决策 |
| 用户描述清晰但有细节未定 | A：纯对齐 | 方向明确，只需补齐决策 |
| 用户说"我想做 X"且无参考 | A：纯对齐 | 有想法，需细化 |
| 长期项目、术语混乱、多人协作 | C：建模 | 需要统一语言 |
| 不确定 | 默认 A | 走不通再切 B |

### 模式 A：纯对齐（用户已经有明确想法）

1. 逐个提问，一次只问一个问题
2. 每个问题附上推荐答案
3. 按以下顺序逐项确认，每项确认后才进入下一项：
   - **目标**：做什么、成功标准是什么
   - **方案**：技术选型、架构方向
   - **范围**：包含什么、明确不包含什么
   - **约束**：时间/资源/兼容性限制
   - **验证**：怎么确认做完了、验收标准
   全部确认后输出需求摘要
4. 输出：清晰的需求描述

### 模式 B：调研 + 对齐（用户提供了参考或需要调研）

1. 先读所有提供的 URL/文件/参考材料
2. 提炼关键信息，总结发现
3. 🔴 CHECKPOINT · 调研结果确认：
   ```
   调研摘要：
   - 关键发现：<要点列表>
   - 可行方案：<推荐 + 备选>
   - 待确认项：<标注不确定的信息>
   
   调研结果 OK？确认后进入提问对齐。(Y/n)
   ```
4. 用户确认后，基于调研结果提问（同样一次一个）
5. 输出：调研摘要 + 清晰的需求描述

### 模式 C：建模（长期项目的术语对齐）

当项目持续时间长、术语混乱时：
1. 维护 `CONTEXT.md` — 项目的领域术语表
2. 用户用词与术语表冲突时立即指出
3. 模糊术语提出精确替代
4. 关键决策记录到 `docs/adr/` 作为 ADR

## 对齐规则

- **🔴 严格遵守：一次只问一个问题，等用户回复再继续**
- 每个问题附推荐答案
- 能从代码库找到答案的问题，去代码库里找，不问用户
- 用户说"你决定"时，用推荐答案并告知
- M级任务：限制 3 个问题以内
- L级任务：不限问题数，走完整决策树

## 调研规则

- 用户给的 URL 全部读取，提炼要点
- PDF 文件同样读取，提取关键内容
- 技术方案需要调研时，搜索最新文档和实践
- 调研结果用结构化格式呈现（要点列表、对比表格）
- 不确定的信息标注"待确认"

## 产物

对齐完成后，输出一份结构化摘要：
```
## 需求摘要
- 目标：<一句话说清楚要做什么>
- 约束：<技术/时间/资源限制>
- 范围：<包含什么、不包含什么>
- 关键决策：<调研中确定的技术选型/方案>
- 参考：<调研中使用的 URL/文档>
```

## 失败处理

| 触发条件 | 一线修复 | 仍失败兜底 |
|---------|---------|-----------|
| URL 读取失败/403/超时 | 换 web_extract 或 curl 重试 | 告知用户 URL 不可读，让用户手动粘贴内容 |
| 调研结果互相矛盾 | 列出矛盾点，标注来源可信度 | 以官方文档/最新来源为准 |
| 问了太多轮用户不耐烦 | 压缩剩余问题，合并能合并的 | 用推荐答案填充未决项，告知用户 |
| 术语表冲突无法调和 | 展示两个定义让用户选 | 用更精确的那个，记录 ADR |
| M 级任务问了 3 个问题还没对齐 | 升级为 L 级，继续问 | 用已有信息先推进，边做边对齐 |
| 用户长时间不回复（>2轮等待） | 总结已有信息，给出推荐方案 | 用推荐方案填充未决项，告知用户"按推荐执行，有异议随时说" |
| 调研结果和用户预期完全不符 | 列出差异点，标注哪些是事实哪些是推测 | 以调研证据为准，建议用户调整方向 |
| 用户回答自相矛盾 | 标记矛盾点，列出两条路径的后果 | 问用户"以哪个为准？"，不自行猜测 |

## 🔴 CHECKPOINT · 需求确认

对齐完成后：
```
需求摘要：
- 目标：<一句话>
- 范围：<包含/不包含>
- 关键决策：<技术选型>

确认后按此执行？(Y/n)
```

用户确认后，自动进入下一步，不再停顿：
- M-phased → 加载 `dj-tdd` skill
- L 级 → 加载 `output` skill（PRD 完成后还有一次确认）

## 衔接（由 dispatch 驱动）

grill 完成后，回到 dispatch 流水线的下一步：
- **M-phased**：用户确认后，加载 `dj-tdd` skill 按切片实现
- **L 级**：用户确认后，加载 `output` skill 写 PRD+设计文档

M-simple 不经过 grill，dispatch 直接实现。

## 反例

| ❌ 不要做 | ✅ 正确做法 |
|---|---|
| **一次问多个问题（最常见的错误）** | **严格一次一个问题，等回复再继续** |
| 没调研就问"你想用什么方案" | 先调研再推荐 |
| 用户给了 URL 不读就开始问 | 先读完再提问 |
| M 级任务问 10 个问题 | M 级限制 3 个问题 |
| 术语混乱不指出 | 立即纠正并记录到 CONTEXT.md |
| 调研结果不确定却当作定论 | 标注"待确认" |
| 对齐完了不输出需求摘要就直接写代码 | 必须先输出需求摘要，等用户确认再继续 |
| 每个问题只问"你想怎么做？"不附推荐答案 | 每个问题必须附推荐答案，降低用户决策负担 |
| 用户说"你决定"还在继续追问 | 用推荐答案填充，告知用户并继续推进 |
| 调研只读一个来源就下结论 | 至少对比 2 个来源，不确定的标注"待确认" |
