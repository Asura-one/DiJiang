# Absorb: 选择性吸收融合

## 这个流程要解决什么问题

做产品时经常看到其他项目有值得借鉴的东西——一个优雅的交互方式、一套舒服的配色、一个巧妙的数据结构。但"借鉴"和"复刻"是两回事：

- **复刻**（`dj-remix`）是把目标整个抄过来再做差异化，目标是一个完整的站点/App，产出是一个新的 standalone 项目
- **吸收融合**（本 skill）是以自有项目为主体，有选择地提取外部目标中的特定元素，**增量地融入**已有代码库

两者做错的风险也不同。吸收融合的主要风险不是"漏功能"，而是：

1. **风格冲突** — 吸收的元素和自有项目的视觉/代码风格不统一
2. **架构不匹配** — 外部模式在自有项目的技术栈或架构下不适用
3. **过度吸收** — 吸收太多，项目变成各种风格的拼凑品
4. **选择性偏误** — 只看到好的部分，忽略了那个设计在源项目中的上下文约束（比如一个交互看起来简洁，是因为后面有强大的后端支持）

所以吸收融合不能是"看到喜欢就搬过来"，而要走完这五步：

1. **摸底** — 搞清楚自有项目现状，明确外部目标
2. **拆解与筛选** — 系统化拆解目标，不是凭第一印象挑，而是有框架地筛选
3. **融合方案** — 每一候选元素设计适配方案，评估冲突
4. **落地实现** — 一条一条实现，不一次改太多
5. **最终验证** — 检查一致性、回归、用户确认

## 强度等级

| 等级 | 适用场景 |
|------|----------|
| **lite** | 只吸收 1-2 个明确元素（比如"借鉴那个项目的配色"），快速落地，跳过 Phase 1 的完整拆解 |
| **full**（默认）| 标准吸收融合流程，完整走过全部 Phase |
| **ultra** | 多目标融合、架构级吸收（比如数据模型、状态管理模式的借鉴），增加冲突分析与替代方案评估 |

## Phase 0：摸底

在拆解任何外部目标之前，先搞清楚自有项目的现状。这一步不能跳过——不知道自己的项目长什么样，就不可能设计出能匹配的融合方案。

**自有项目快照**

1. 浏览项目结构（目录布局、技术栈）
2. 现有组件体系（使用了哪些 UI 组件库、自定义组件）
3. 现有的设计语言/视觉规范（如果有设计规范或主题配置，读一遍）
4. 当前代码风格和命名约定
5. 现有架构模式（状态管理、数据流、路由方式）
6. 对当前项目的"痛点"或"想要改进的方向"（如果有），记录下来

产出：自有项目快照文档（记录在对话或文件中，不做成独立文件，但必须记录下来供后续 Phase 2 参考）。

**外部目标确认**

- 有可访问的 URL：用 `fetch_content` 抓取关键页面
- 有截图或设计稿：用 `read` 逐张查看
- 只给关键词/产品名：确认指向的具体产品，用 `web_search` 找官网或截图
- 多个目标时，分别确认每个目标的访问方式

**吸收范围确认**

问用户明确的问题：
- 具体看中这个目标的什么？（整个视觉风格？某个交互组件？数据展示方式？）
- 吸收到什么程度？（完全照搬再调风格，还是只借鉴思路自己重新实现？）
- 有没有已经有想法的融合方向？（或者愿意看建议？）
- 只吸收一个目标还是从多个目标中分别提取元素？

## Phase 1：拆解与筛选

对每个外部目标，从吸收融合的视角做系统化拆解，而不是复刻视角。拆解维度不同——不关心"这个页面有哪些功能需要全量复制"，而是关注"有什么值得借鉴的元素"。

拆解维度参考 `references/absorption-checklist.md`，按目标类型选择对应维度。每个候选元素记录一张卡片：

```
## 候选元素卡片

来源：[目标名称]
元素类型：[UI 组件 / 交互模式 / 视觉语言 / 数据模型 / 架构模式 / 微交互]
描述：
吸収价值：[为什么这个值得吸收？比自有项目现有哪些优势？]
适配成本：[融入自有项目需要多大改动？]
冲突风险：[与自有项目现有风格/架构的冲突程度：高/中/低]
是否吸收：[是 / 否 / 待讨论]
理由：
```

筛选原则：
- **吸収价值 > 适配成本 + 冲突风险** 才吸收
- 同类元素来自多个目标时，优先吸收最有价值的，避免同类元素打架
- 技术栈不兼容、架构层冲突明显的，标记为"不适合吸收"并说明原因，而不是硬搬
- **如果拆解过程中发现信息缺失**（截图看不到、URL 抓不到的动态交互），记下"未知/待确认"，不在 Phase 2 之前脑补

## Phase 2：融合方案

对 Phase 1 中标记为"是"或"待讨论"的候选元素，逐个设计融合方案。这是"吸收融合"与"完整复刻"的关键区别——不是照搬，而是适配到自有项目。

每个元素的融合方案包含：

```
## 融合方案：[元素名称]

### 适配分析
- 源实现方式：[目标中如何实现的]
- 自有项目现有方案：[目前有的 / 没有的情况]
- 适配方式：[直接搬代码再改 / 参考思路重新实现 / 部分参考 + 部分自定义]
- 改动范围：[涉及的文件和模块]

### 冲突解决
- 风格冲突：[与现有视觉风格是否一致？如何调和？]
- 架构冲突：[与现有数据流/状态管理是否兼容？]
- 命名冲突：[标识符与现有代码是否冲突？]

### 实现策略
- 改动方案：[具体怎么改]
- 新代码 vs 改现有代码：[是新增组件还是修改现有组件？]
- 依赖影响：[新增依赖？现有依赖的版本冲突？]

### 验证方式
- 验证清单：[怎么确认这次改动是正确的]
```

多目标融合时的特殊处理：
- 从不同目标吸收的元素放到一起后，可能存在风格冲突 —— 此时以自有项目为主体，调整外来元素去匹配自有风格，而不是让自有风格去匹配外来元素
- 如果两个目标中对同类元素有不同方案，选择与自有项目更兼容的那个，或设计一个吸收两者优点且与自有项目一致的新方案

**🔴 CHECKPOINT · 🛑 STOP：展示所有融合方案给用户确认。** 用户说"继续"再进入 Phase 3。用户可能在此阶段调整吸收范围、修改方案、或决定不吸收某些元素。

## Phase 3：落地实现

按 Phase 2 确认后的融合方案，逐条实现。核心原则：**一条一条做，做完一条验证一条，不堆在一起验收**。

- 每次只实现一个候选元素的融合
- 实现前确认该元素不依赖其他未实现的元素（如有依赖，先实现被依赖的）
- 每完成一条，对照 Phase 2 中该元素的验证方式做快速验证
- 涉及视觉/样式的改动，参考 `dj-design` skill 的设计规范中与自有项目一致的部分
- 遇到实现困难（技术限制、冲突超出预期），暂停并评估：降低适配要求 / 换实现方式 / 放弃该元素
- 每完成一个元素，更新代码和验证记录

**如果在 Phase 3 发现 Phase 2 的方案有问题**（比如实际代码中发现未预料的冲突），不要硬撑原方案，回到 Phase 2 调整方案再继续。

## Phase 4：最终验证

全部元素实现完成后，做整体验证：

1. **一致性检查**：新增/改动的代码是否和自有项目现有代码风格一致（命名、结构、注释风格）
2. **视觉一致性**：新引入的视觉元素是否与自有项目整体视觉风格协调（如果有主题配置，确认新元素正确使用了主题变量而不是硬编码颜色）
3. **回归检查**：自有项目原有功能是否不受影响（跑测试、编译检查、手动检查受影响页面）
4. **多目标融合检查**：如果从多个目标吸收了不同元素，它们之间是否协调，会不会看起来像拼凑的
5. **差异清单**：交付时清楚列出"从哪里吸收了哪些元素、做了什么适配改动"，方便项目成员了解
6. **用户确认**：展示成果，确认交付

## Hard Rules

1. 必须走完全部 5 个 Phase，不能跳过 Phase 0（摸底）或 Phase 2（融合方案设计）
2. 拆分清单和融合方案必须书面化（对话或文件），不能只在脑子里
3. Phase 2 的方案必须经过用户确认才能进入 Phase 3
4. 遇到技术限制或不兼容，必须明确告知用户原因和替代方案，不能悄悄简化
5. 不能以"借鉴"为名完整复刻整个页面或功能——那是 `dj-remix` 的范畴，不在本 skill 中做
6. 吸收改造不能破坏自有项目的风格/架构一致性——每引入一个外来元素，都要确认它和现有部分匹配
7. 多目标吸收时，自有项目是主体，外来元素适配自有风格，反之不行
8. 每实现一个元素就验证一个元素，不堆到 Phase 4 才一次性检查

## Gotchas

| Gotcha | 后果 | 预防 |
|---|---|---|
| 跳过 Phase 0 直接拆解目标 | 融合方案与自有项目不匹配，返工 | 先摸底再拆解 |
| 看到想借鉴的部分就忽略上下文约束 | 搬过来后发现缺少依赖条件，不能用 | 拆解时同时关注依赖条件 |
| 一次吸收太多元素 | 项目风格碎片化 | 一次吸收 3-5 个元素后做一致性检查 |
| 只从一个目标吸收，忽略自有项目风格 | 新元素像外来的补丁 | 每元素做冲突分析 |
| 懒得改方案直接搬代码 | 风格不统一，代码难以维护 | 融合方案阶段做好适配设计 |
| 从多个目标吸收同类元素 | 同类功能有多种实现风格 | 筛选时优先兼容性好的，或设计统一方案 |
| Phase 3 遇到冲突不回头调整方案 | 硬撑导致半成品 | 冲突超预期就暂停，回到 Phase 2 调整 |
| 用户说"借鉴这个"但不指明具体元素 | 吸收范围太大，产出无焦点 | Phase 0 时追问清楚具体看中什么 |
