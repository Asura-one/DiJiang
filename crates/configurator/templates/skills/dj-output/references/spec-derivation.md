# Spec 推导原则

> 不面试用户。只从已有对话上下文中综合出 spec。

## 核心约束

- **不提问**：不要问用户"你想要什么功能？""这个需求怎么定义？"。答案已经在之前的对话中。
- **只综合**：从对话历史、PRD、design doc、issue 描述中提取结构化定义
- **标记缺口**：对话中未覆盖的部分标注 `[TODO: 待确认]`，而不是追问用户
- **保持可追溯**：每个 spec 条款标注来源（"来自用户描述"、"来自 design doc"）

## 适用场景

- 从 dj-grill 或讨论会话转向实现时
- 写 PRD/design doc 时
- 拆票（to-tickets）前

## 不适用场景

- 用户明确要求"帮我写下 PRD，我先告诉你要什么"——这时用户主动要写，提问是合法的
- 纯技术说明文档——不需要需求推导
