# Bug 诊断反馈环

从 Matt Pocock 的 `diagnosing-bugs` skill 提取的核心原则：**在碰代码之前，先建立 pass/fail 反馈信号。**

## 原则

> 花不成比例的时间在 Phase 1（建立反馈环）上。只有有了稳定的反馈信号，才能开始分析代码。

## 流程

### Phase 1：建立反馈环（占诊断总时间 >40%）

目的：有一个可以稳定触发"问题出现了 / 问题没出现"的开关。

1. 找到稳定复现路径
   - 命令行：`npm test` / `cargo test` / `python -m pytest <specific test>`
   - 手动：精确到每一步操作、输入值、环境条件
   - 不可能稳定复现时：建立"问题存在"的证据链（日志、监控、用户报告模式）

2. 简化反馈环
   - 去掉无关步骤，只保留触发问题的最短路径
   - 确认"问题在 / 问题不在"的区分度足够

3. 编写复现测试（如果可行）
   - 先写一个会 FAIL 的测试
   - 确认：改完全无关的代码，测试不应该 pass

**Phase 1 完成标准**：能在 30 秒内确认"问题还在不在"。

### Phase 2：分析代码（占 30%）

1. 从反馈环出发，反向追踪代码路径
2. 用二分法缩小范围（git bisect、代码注释法、输入参数裁剪）
3. 定位到具体的函数/变量/逻辑行

**Phase 2 完成标准**：能用一句话说清根因。

### Phase 3：修复验证（占 20%）

1. 一个改动一次验证
2. 每次改完后回到 Phase 1 的反馈环确认
3. 修复后添加 regression 测试

### Phase 4：回退处理（占 10%）

1. 3 次连续修复失败 → 停止并汇报
2. 升级：补充日志、检查外部依赖、查类似 issue
3. 实在阻塞时，输出"当前已知 + 不确定 + 建议"并问下一步
