## Outcome Contract

| 项目 | 内容 |
|---|---|
| **Outcome** | 代码模式分析报告 |
| **Done when** | 扫描维度确认 → 全仓扫描完成 → 报告输出 |
| **Evidence** | 扫描日志、grep 结果 |
| **Output** | 结构化模式报告（可复用抽象 + 反模式 + 需要改进的领域） |

# Pattern: 模式识别

从历史修复和代码模式中学习，发现可复用的抽象和需要改进的反模式。只报告，不修改。

## 工作流

### 1. 确定扫描范围

确认关注的维度：
- **重复代码** — 类似结构出现 3 次以上
- **历史修复模式** — 相似 bug 反复出现
- **术语不一致** — 同一概念多个命名

### 2. 重复模式扫描

搜索项目中重复出现的代码结构、类型定义或逻辑片段。报告出现位置和重复程度。

### 3. 历史修复分析

从 git log 中提取近期修复，寻找模式和趋势：
```bash
git log --oneline -30
```
检查是否有：
- 同类问题反复修复
- 修复范围是否越来越广
- 根因是否未被真正解决

### 4. 抽象建议

如果确实存在可复用模式，给出建议：
- 新抽象的位置（哪个文件/模块）
- 抽象边界（包含什么、不包含什么）
- 迁移成本估算（影响多少调用方）

### 5. 🔴 建议门禁

```text
模式：<描述>
证据：<出现位置 N 处>
建议：<抽象方向>
成本：<约改动 M 处>
价值：<消除重复 / 统一概念 / 减少 bug>
```

## 边界

- 不修改代码（只报告）
- 三次重复以下不视为模式
- 不评估非代码模式（流程、团队习惯）

## 反例

| ❌ 不要做 | ✅ 正确做法 |
|---|---|
| 两次重复就建议抽离 | 至少三次再谈抽象 |
| 不给证据就提建议 | 列出所有出现位置 |
| 建议抽象但不评估成本 | 估算影响范围 |

## Hard Rules

1. 只识别已有模式，不创造新模式
2. 发现模式必须附原始代码引用，不凭空描述
3. 反模式必须区分"确定"和"疑似"
4. 不在本 skill 中修改代码——只报告

## Gotchas

| Gotcha | 后果 | 预防 |
|---|---|---|
| 发现模式就建议抽象 | 没有评估收益/成本 | 发现 + 评估成本一起出 |
| 描述不引代码 | 无法验证 | 发现必须带文件+行号 |
| 把偏好当反模式 | 报告可信度低 | "疑似" vs "确定"要分清 |
