# Prototype: 造废品验证设计

造能跑的废品来回答问题——不是做产品。

## 什么时候用

- 不确定技术方案能不能走通
- 需要看实际效果才能决策
- 有两个选项，不确定选哪个
- 预研新技术/库

## 规则

- **目标是回答问题**，不是写可维护代码
- **写完就扔** — 验证完不合并到主分支，结论记下来就行
- **不要自动化** — 没有 lint / test / CI / 错误处理
- **时间盒** — 最多 2 小时，到时间就汇报结论

## 工作流

### 1. 定义问题

```text
要回答的问题：<一个可以验证真伪的陈述>
验证方式：<手动操作 / 跑一段代码 / 看输出>
成功标准：<什么结果算"方案可行">
```

### 2. 写最小验证代码

- 写能回答问题的最小代码量
- 硬编码 > 配置；手动 > 自动化；控制台 > UI
- 不做错误处理、不做边界校验、不做类型安全

### 3. 运行并记录

- 运行验证，记录输入和输出
- 回答一开始定义的问题
- 写一句结论：行 / 不行 / 有条件行（条件是什么）

### 4. 清理

- 原型代码不提交
- 结论记录到 `.dijiang/` 或告诉用户
- 问：下一步是不是应该用 `dj-implement` 做正式版？

参考 `references/prototype-patterns.md` 查看原型验证模式。

## 反例

| ❌ 不要做 | ✅ 正确做法 |
|---|---|
| 给原型加测试 | 原型的目标是回答问题 |
| 写可维护的代码 | 写最少代码，答完就扔 |
| 原型代码合并到 main | 原型是废品，不保留 |
| 超过 2 小时还在调 | 到时间就汇报 |
