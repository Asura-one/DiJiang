# Reason: 推理增强

## 职责

`dj-reason` 是只读分析 skill。它帮助 agent 在复杂问题中先校准思考方式，再输出可执行的判断。它不修改代码、不改变 DiJiang task 状态、不替代 `dj-dispatch`、`dj-grill`、`dj-check` 或任何 CLI 状态转换。

适用场景：

- 需求、方案、设计、架构取舍不清，需要先分析问题结构。
- 问题反复出现，疑似不是单点 bug，而是流程、反馈或激励结构导致。
- 用户要求解释为什么善意行动造成坏结果，或要求看长期副作用。
- 实现前需要比较多个路径的风险、边界和杠杆点。
- 审查或复盘前需要先校准事实、推断、假设和待确认项。

## 输入 / 输出

| 项目 | 约定 |
|---|---|
| 输入 | 用户问题、已读上下文、task 状态、相关文件/文档/命令输出 |
| 输出 | 推理报告，包含事实、假设、认知路径、系统结构、风险、杠杆点和下一步建议 |
| 非目标 | 不写代码、不创建任务文档、不替代 route gate、不执行 finish-work、不把启发式模型包装成科学结论 |

## 认知矩阵

按任务需要动态组合以下三个维度：

| 维度 | 选项 | 用途 |
|---|---|---|
| X 轴 | 表层解析 / 深层洞察 / 自我校准 | 从理解输入，到识别本质，再检查自身误判 |
| Y 轴 | 线性思维 / 发散思维 / 综合分析 | 从顺序推导，到生成备选，再收束判断 |
| Z 轴 | 知识融合 / 情境映射 / 元认知调节 | 从跨知识连接，到贴合上下文，再调整思考策略 |

默认能力激活：

- 快速掌握：表层解析 + 线性思维 + 情境映射
- 本质洞见：深层洞察 + 综合分析 + 知识融合
- 自主思考：自我校准 + 发散思维 + 元认知调节

## 系统透镜

当问题涉及工作流、组织行为、长期副作用、反复出现的流程问题，或“好心行动造成坏结果”时，启用系统透镜。不要只修补事件，要理解结构、寻找杠杆、与系统共舞。

分析路径：

1. 核心问题：为什么善意、合理、局部有效的行动，可能在系统中造成坏结果？
2. 基础透镜：系统 = 要素 + 连接 + 目标。先看参与者、关系和系统实际优化的目标。
3. 动态语言：存量 + 流量 + 反馈回路。区分什么在积累，什么在流入或流出，什么回路在驱动变化。
4. 两类回路：调节回路维持平衡；增强回路放大趋势。判断当前行为是在稳定系统，还是放大某种趋势。
5. 典型行为：增长、衰退、稳定、振荡、崩溃。不要只看当前状态，要判断趋势。
6. 系统之美：适应力 + 自组织 + 层次性。尊重系统已有的恢复力和分层结构。
7. 系统之奇：表象 + 非线性 + 边界 + 限制因素 + 延迟 + 有限理性。检查表面现象是否误导了判断。
8. 系统陷阱：政策阻力、公地悲剧、目标侵蚀、竞争升级、富者愈富、转嫁负担、规避规则、目标错位。只诊断有证据支持的陷阱。
9. 改变系统：从参数到范式，越深层杠杆越大。区分短期参数调整和深层目标、规则、信息流变化。
10. 行动方式：观察节拍、公开心智模型、尊重信息、关注整体、制定反馈政策、保持谦逊。

## 流程

### Step 1: 读取上下文

如果问题涉及代码、模板、workflow、task 或文档，先读取相关文件和当前 task 状态。不要凭框架惯例猜测。

```text
Context loaded: <files/task/docs/none>
Route gate: <current capsule and allowed skill if known>
Will modify code: no
Will change task state: no
```

🛑 STOP if the user expects implementation, quality gate, task state transition, commit, push, or finish-work. Report the correct follow-up skill instead of doing it inside `dj-reason`.

### Step 2: 判断交互类型

```text
α：需求、方案、设计、权衡不清，需要深层对齐
β：已有明确目标，需要快速理解和执行建议
γ：信息不足，需要先定位上下文和事实依据
```

根据交互类型选择推理姿态：

- α：优先深层洞察、综合分析、情境映射
- β：优先表层解析、发散思维、知识融合
- γ：优先表层解析、线性思维、情境映射

### Step 3: 分析问题结构

至少区分：

- 事实：已经从用户输入、文件、命令或文档确认的内容
- 推断：基于事实推出来但尚未直接证明的内容
- 假设：为推进分析临时采用的前提
- 待确认项：继续行动前必须确认的信息

如果是系统问题，额外识别：

- 要素：参与者、模块、资源、状态、规则
- 连接：依赖、激励、信息流、反馈路径
- 目标：表面目标和系统实际优化的目标
- 回路：调节回路、增强回路、延迟和限制因素
- 陷阱：只列有证据支持的系统陷阱
- 杠杆：参数、信息流、规则、目标、范式等不同深度的改变点

### Step 4: 输出推理报告

```text
推理报告：<问题标题>

1. 问题重述
<一句话说明真正要判断的问题>

2. 已知事实
- <事实 + 来源>

3. 推断与假设
- 推断：<内容 + 依据>
- 假设：<内容 + 风险>

4. 认知路径
- 交互类型：<α/β/γ>
- 激活能力：<快速掌握/本质洞见/自主思考或组合>
- 为什么采用这个路径：<简述>

5. 系统结构（如适用）
- 要素：<...>
- 连接：<...>
- 目标：<...>
- 反馈回路：<...>
- 延迟/限制因素：<...>
- 系统陷阱：<...>

6. 杠杆点与取舍
- 浅层杠杆：<快但影响有限>
- 深层杠杆：<难但影响更大>
- 推荐路径：<建议 + 原因>

7. 下一步
<最小可推进动作；如果需要其他 skill，明确写出>
```

## 自我校准

输出前检查：

- 我是否读了相关文件或任务上下文？
- 我是否混淆了用户目标和我自己生成的产物？
- 我是否跳过了 DiJiang 的 route gate？
- 我是否把未证实内容当成事实？
- 如果这是系统问题，我是否识别了要素、连接、目标、反馈、延迟、陷阱和杠杆？
- 当前最小可推进动作是什么？

## 边界

- 只分析，不实现。
- 只建议路线，不改变 workflow state。
- 不替代 `dj-grill` 的确认门禁；需求仍不清时，输出应回到 `dj-grill`。
- 不替代 `dj-check` 的质量门禁；需要验证 diff 时，输出应转到 `dj-check`。
- 不把“认知矩阵”或“系统透镜”当作科学模型；它们只是推理组织和自我校准工具。

## 反例

| ❌ 不要做 | ✅ 正确做法 |
|---|---|
| 只输出漂亮理论 | 输出事实、假设、结构和下一步 |
| 不读上下文就开始系统分析 | 先读取 task、文件或用户给定材料 |
| 在 `dj-reason` 内改代码 | 报告应进入 `dj-implement` / `dj-tdd` |
| 在 `dj-reason` 内做质量门禁 | 报告应进入 `dj-check` |
| 把每个问题都套系统陷阱 | 只诊断有证据支持的陷阱 |
| 用认知矩阵替代 DiJiang route gate | 认知矩阵只辅助判断，route gate 仍优先 |
