## Outcome Contract

| 项目 | 内容 |
|---|---|
| **Outcome** | 结构化推理分析 + 推荐方向 |
| **Done when** | 认知矩阵各维度分析完成，决策过程显式化 |
| **Evidence** | 推理记录（各维度分析过程） |
| **Output** | 结构化推理报告 + 推荐方向及理由 |

# Reason: 推理增强

用结构化思考分析复杂判断和架构取舍。不是替代决策，而是让决策过程显式化。

## 认知矩阵

分析方案时，从这些维度展开：

| 维度 | 问题 |
|---|---|
| 目标 | 这个方案要解决什么问题？ |
| 约束 | 哪些是不可变的硬约束？ |
| 取舍 | 选了这个，放弃了什么？ |
| 风险 | 什么情况下这个方案会失败？ |
| 成本 | 实现和维护的成本估计？ |

## 模式

根据问题深度选择分析模式：

| 模式 | 何时用 | 输出 |
|---|---|---|
| **轻量级** | 日常决策：命名、API 签名、简单取舍 | 一句话推荐 + 一条理由 |
| **评估模式** | 评估已有方案：这个设计好吗？ | 结构化评估（优点/缺点/风险） |
| **分类模式** | 问题太复杂：路由到正确流程 | 分类结果 + 推荐下一个 skill |

详见 `.dijiang/references/design-modes.md`。

## 系统透镜

从系统角度分析影响：

1. **局部 vs 全局** — 这个改动对当前模块好，但对整个系统呢？
2. **短期 vs 长期** — 现在快但以后慢？还是现在慢但以后快？
3. **可逆 vs 不可逆** — 错了能回滚吗？数据迁移、API 发布是不可逆的

## 流程

1. 陈述要分析的决策
2. 列出候选方案（至少 2 个）
3. 用认知矩阵分析每个方案
4. 用系统透镜评估影响面
5. 给出推荐和理由

## 边界

- 不做执行——只分析、推荐、报告
- 不给"完美方案"——所有方案都有取舍
- 如果两个方案都很差，说实话

## Hard Rules

1. 不做执行——只分析、推荐、报告
2. 不给"完美方案"——所有方案都有取舍
3. 如果两个方案都很差，说实话
4. 认知矩阵每一格都要填——不能跳过难分析的维度

## Gotchas

| Gotcha | 后果 | 预防 |
|---|---|---|
| 只讲优点不讲代价 | 失真分析 | 每个方案必须有取舍分析 |
| 跳过难分析的维度 | 分析不完整 | 所有维度都要填，难的就写"不确定" |
| 追求完美的方案 | 浪费时间的分析 | 限制在合理深度，给出推荐 |
