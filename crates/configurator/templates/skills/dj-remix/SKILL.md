# Remix: 网站/App 复刻与差异化改造

## 这个流程要解决什么问题

直接照着一个网站/App"边看边写代码"，几乎必然会漏东西——人（以及模型）的注意力天然会被主视觉和主流程吸引，次要功能、边缘状态（空状态/加载中/报错/禁用）、隐藏交互很容易被忽略，最后交出的是一份"看起来像"但功能覆盖不全的阉割版。

同时，如果复刻得"太像"，原样照搬，既可能有版权/商标风险，也没有体现使用者自己的产品调性。

所以复刻不能是"看一眼、写一遍"，而要走完这五步：

1. **确认范围与技术栈** —— 明确输入、复刻边界、产出形式
2. **全面拆解** —— 把目标像拆机器一样拆成清单，而不是凭第一印象开始写
3. **和用户核对清单** —— 补齐盲区，这是防止漏功能最关键的检查点
4. **按清单搭建** —— 功能对等优先，遇到做不到的地方要明说，不能悄悄简化
5. **差异化改造 + 最终自查** —— 这一步是必做项，不是可选项

不要跳过第2、3步直接开始写代码，也不要跳过第5步的差异化改造——这两处正是"复刻不完整"和"复刻没特色"这两个问题各自的解法，缺一个都会让用户不满意。

## Phase 0：确认范围与技术栈

开始前先搞清楚三件事，模糊地开始只会导致返工：

**输入来源**
- 有可访问的 URL：用 `fetch_content` 抓取，不要只看首页——列出能找到的所有关键路由/页面分别抓取。单页应用可能需要多抓几个状态（比如登录前/登录后能看到的内容不同）。
- 有截图或设计稿：用 `read` 逐张仔细看。**截图天然是静态快照，看不到 hover、动效、报错提示、加载状态**——这些要主动向用户问清楚，不要靠猜测补全；猜出来的交互往往就是后续被吐槽"复刻得不对"的地方。
- 只给了一个关键词/产品名，没有URL也没有截图：先确认这个关键词具体指向哪个产品——如果有歧义（同名产品、或者只是一个泛类目描述，比如"一个类似小红书的社区App"这种没点名具体产品的），跟用户确认清楚指的是谁，不要凭第一反应认定。确认后用 `web_search` 找官网/应用商店页面，能拿到URL就回到第一种情况用 `fetch_content` 抓；抓不到网页内容（比如纯原生App）或者没有官网，就用截图搜索找界面截图，再搜几篇产品评测/功能介绍类文章补充截图里看不到的交互逻辑。**关键词是信息量最少的起点，越是这种情况越要主动多做几轮调研，而不是因为信息少就跳过调研直接凭印象开始列清单**——信息不够时最容易脑补，这恰恰是漏功能的重灾区。这一步做的调研本身也是在为 Phase 1 攒素材，不算额外功夫。
- 以上几种混合出现：优先用信息量更大的来源（URL/搜索到的官网）去补全信息量较小的来源（截图/关键词）看不到的部分。

**复刻范围**：整站/整个App，还是某几个核心页面或某条流程？范围越大，Phase 1 的拆解就越要系统化，不能因为范围大就走捷径漏项。

**是从零开始，还是本地已有一个复刻项目**：如果本地已经有一个复刻项目、但复刻得不全面，不要直接当成"从零开始"处理——跳到下面「特殊情况：本地已有一个不完整的复刻项目」，按那里的调整方式走。

**技术栈**：默认建议用 React + Tailwind 组件产出，但正式动手前要跟用户明确确认一次（除非用户当次任务里已经说清楚了）。原因：
- 有些目标更适合用静态 HTML/CSS/JS（比如内容型/营销型页面，不需要复杂状态管理）
- 用户可能希望尽量贴近原网站/App实际使用的技术栈

不要默认直接开始用 React + Tailwind 写，先确认一句再动手。

## 特殊情况：本地已有一个不完整的复刻项目

这种情况不能当成"从零开始"处理，也不能只看现有代码就假定复刻范围到此为止。按下面的方式调整流程：

1. **先摸清现有项目的实际情况**：浏览项目结构，看已经实现了哪些页面、组件、功能——这是为了后面做对照，**不是**为了拿它来确定复刻范围的边界。
2. **仍然要独立、完整地走一遍 Phase 1**：针对**原始复刻目标**（原网站/App本身）重新做一次完整拆解，不要依赖"现有项目已经做了什么"去反推范围够不够。这是最容易踩的坑：如果只在现有项目的认知边界内查漏补缺，那些原本就被忽略、从未被实现过的功能永远不会被发现——因为它们压根不在"已有项目"这个参照系里。这也正是当初复刻不全面的原因，重复同样的方式只会得到同样的结果。
3. **做一次对照**：把 Phase 1 的完整清单和现有项目逐项比对，标出三种状态——已完整实现 / 部分实现（阉割版，比如只做了默认状态没做报错和空状态）/ 完全没做。
4. **Phase 2 呈现的是这份差距清单**，而不是从头开始的完整清单——问用户这些缺失项是否都要补，还是当初就是有意精简、不需要补。
5. **Phase 3 只补差距**：已经做得完整的部分不用动，只实现缺失/阉割的部分；补的时候要跟现有代码的写法、组件结构、命名习惯保持一致，除非用户明确想重构。
6. **Phase 4 要先检查现有项目有没有做过差异化改造**：如果完全没做过，仍然要走这一步，不能因为这次任务是"补全"就跳过；如果已经做过，新补的部分要匹配现有的差异化风格（配色、版式语言），避免新旧部分气质不统一、看起来像拼凑上去的。
7. Phase 5 照常做最终自查，交付说明里注明这次是"补全"性质，列清楚哪些是新增的、哪些是原来就有的。

## Phase 1：全面拆解目标（决定复刻是否完整的核心步骤）

把目标当成一台机器来拆，而不是"看一眼记个大概"。在动手写任何代码之前，先产出一份**书面的清单**（写在对话里或列成清单都可以，但一定要写出来，不能只放在脑子里）——只有写出来，Phase 3 的"逐项核对"才有东西可对，Phase 2 用户核对时才有东西可看。

清单至少要覆盖这些维度（完整的分类和逐项提示见 `references/decomposition-checklist.md`，里面按电商/SaaS后台/内容站/社交/营销页/移动App等不同类型都有专项补充，动手列清单前先扫一遍再开始）：

1. **信息架构**：完整的页面/路由清单，导航层级
2. **页面级拆解**：每个页面从上到下有哪些区块，区块的顺序和层级关系
3. **组件清单**：所有会被复用的UI组件，以及每个组件的**各种状态**（默认/hover/激活/禁用/加载中/空状态/报错/聚焦）——这一项最容易漏，一个按钮不是只有"能点"这一种状态
4. **内容与数据结构**：每个内容区块展示的数据字段是什么（比如一张商品卡片需要标题/价格/图片/评分/标签）
5. **交互与行为**：点击、拖拽、动效过渡、表单校验规则、搜索/筛选/排序逻辑、快捷键
6. **关键用户流程**：注册、登录、下单、引导流程等多步骤流程，以及每一步的成功/失败/边界情况
7. **响应式行为**：断点、移动端和桌面端的布局差异
8. **全局元素**：header/footer/导航、弹窗/toast通知系统、深色/浅色模式
9. **容易被漏掉的"次要功能"**：分页、无限滚动、空状态、404页面、tooltip、面包屑、cookie提示条、骨架屏加载、无障碍细节

拆解过程中如果发现有信息缺失（比如截图没覆盖到、URL抓不到的动态内容），直接记下"未知/待确认"，留到 Phase 2 问用户，不要自己脑补。

## Phase 2：和用户核对清单，补齐盲区

把 Phase 1 的清单原样呈现给用户，明确问：

- 有没有遗漏的页面/功能/状态？
- 清单里有没有其实用户不需要的部分？（这是用户主动做的精简，跟"复刻遗漏"是两回事，要注意区分，并在 Phase 5 交付说明里写清楚是哪种情况）
- 顺便再确认一次 Phase 0 的技术栈选择

这一步不能省，即使任务看起来很急。原因很直接：与其复刻做完了才发现漏了东西要返工，不如现在花两分钟对齐一次——这是整个流程里性价比最高的一步。

## Phase 3：按清单搭建，功能优先，不做隐性阉割

- 涉及视觉/样式的部分参考 `dj-design` skill 的设计规范
- 严格按 Phase 2 确认后的清单实现，**清单上有的东西都要出现在产出里**，不要因为某项实现起来麻烦就悄悄跳过或简化
- 如果确实有技术限制做不到（比如依赖真实后端数据、第三方支付网关这类），**必须明确告诉用户**："这部分做了简化处理，原因是XX，可选的替代方案是YY"——而不是默不作声地砍掉。这才是"阉割版"问题的根源：往往不是能力不够做不到，而是遇到麻烦就悄悄跳过还不说
- 建议逐个页面/逐个组件推进，做完一部分就对照清单打勾，不要留到最后才一次性检查——那样发现漏项时返工成本最高

## Phase 4：差异化改造（必做，不是可选项）

这一步要同时解决两件事，用户很在意这两件事都要兼顾，不要只做其中一个：
- **去同质化**：避免产出和原网站/App长得一模一样，降低版权/商标风险
- **产品调性**：体现使用者自己的风格，而不是简单的复制粘贴

具体怎么改、改到什么程度，参考 `references/variation-playbook.md`，大方向包括：
- **版式**：调整网格比例、区块顺序、卡片密度/留白节奏，改变hero区域构图（比如左文右图换成分屏或堆叠），导航形式（顶部横向换侧边栏，或反过来）
- **主题**：重新设计一整套配色（不是只换主色，是重新定义背景/文字/边框/强调色的整套对比关系）、字体搭配、圆角/阴影等视觉语言
- **品牌**：用占位的名称/logo，不直接沿用原产品的品牌标识组合

**关键原则**：差异化改的是"表现层"，不是"功能层"。Phase 2 确认过的信息架构、核心流程、组件覆盖范围要保持完整，不能借着"做差异化"的名义顺带砍掉功能——完整性和差异化这两件事互不冲突，也不能互相替代。

建议准备2-3个差异化方向（比如"极简白"/"深色科技感"/"温暖有机风"），列出每个方向分别在"去同质化"和"产品调性"上各自的侧重点，交给用户选，而不是自己直接定一个方向闷头做。

## Phase 5：最终自查与交付

- 逐项对照 Phase 2 确认过的清单，检查最终产出是否都覆盖到了，包括容易被忽略的次要功能和各种状态
- 如果有因技术限制做的简化，交付时清楚列出"哪里简化了、为什么、影响是什么"
- 总结这次做了哪些差异化改动，让用户清楚成品和原版的区别在哪
- 按常规文件交付流程处理（生成的 React 组件/HTML 文件放到 outputs 目录并展示给用户）

## Hard Rules

1. 必须走完全部 5 个 Phase，不能跳过 Phase 2（核对清单）或 Phase 4（差异化改造）
2. 拆解清单必须书面化（对话文本或文件），不能只放在脑子里
3. 遇到技术限制做不到的功能，必须明确告知用户原因和替代方案，不能悄悄简化
4. 差异化改造不能以砍功能为代价——功能完整性和差异化互不冲突
5. 用户流程/敏感功能（注册、登录、支付）必须完整覆盖成功/失败/边界情况

## Gotchas

| Gotcha | 后果 | 预防 |
|---|---|---|
| 跳过 Phase 1/2 直接开始写 | 最终交付严重漏功能 | 列完清单、对齐后再动代码 |
| 截图看不到的交互靠猜 | 复刻交互与原版不一致 | 截图缺失的部分主动问用户 |
| 遇到麻烦悄悄简化而不说明 | 用户拿到阉割版才傻眼 | 做不到就明说，给替代方案 |
| 差异化改造砍了功能 | 交付的功能残缺 | 改完后对照 Phase 2 清单重新过一遍 |
| 关键词输入时跳过调研直接列清单 | 脑补出来的拆解从一开始就偏了 | 信息越少越要多做调研 |
| 补全任务时只在现有项目范围内查漏 | 原来没做过的功能继续被遗漏 | 针对原目标重新拆解，再做对照 |
