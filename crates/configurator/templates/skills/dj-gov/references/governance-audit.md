# 规范审计细则

## 提取可核验约定

读规则链时找能转成检查命令的祈使句：

| 类别 | 核验 |
|---|---|
| 命名约定 | 同级名字清单和例外条款 |
| 必备文件 | 文件存在性与最小内容 |
| 同源声明 | realpath、导入、加载顺序 |
| 安全红线 | tracked files、ignore 规则 |
| 目录纪律 | 根目录 manifest 与白名单 |
| 声明一致 | package/config、服务名、端口 |
| 发布流程 | knowledge receipt → report → user confirmation → cleanup |

## 确定实际规则链

1. 识别当前 agent 和 cwd
2. 从项目根到 cwd 查规则文件、override、fallback
3. 再查用户/组织级规则
4. 记录加载顺序和覆盖关系

## 核验范围

| 范围 | 审什么 |
|---|---|
| 当前项目 | 审当前项目 |
| 跨项目 | 审实际受影响的 upstream/consumer 文档，先说明扩展依据 |
| 整个 workspace | 用户明确授权后逐项目检在在性和同源 |
| 全局配置 | 只读审计死引用、矛盾和加载漂移 |

## 处置等级

### 可直接修

安全、可逆、无外部副作用，且现场规则授权自动修复：
- 规则要求的软链/导入缺失
- `.gitignore` 缺少明文要求的敏感文件模式
- 文档中确认已不存在的死链接
- 同一事实的过期摘要（可由当前代码唯一裁决）

### 先报告再决定

- 目录或项目重命名、文件删除
- 合并不同规则文件且权威无法唯一确定
- 修改范围外项目或全局配置
- 唯一证据在未合并/dirty lane 中

### 只读报告

- 用户限制为"看看/审计"
- 范围外问题、平台生成文件
- 无法验证或需要外部权限的事实

## 规则本身质量

- 死引用：路径、命令、服务、项目是否存在
- 矛盾：上下级规则、override、skill 是否互斥
- 不可执行：大量"保持整洁"但没有判断标准
- 重复：同一红线多处复制导致分叉
- 过载：核心规则被历史叙事挤没
- 无门禁：同一违规第三次出现，建议 check/hook/test 解决
