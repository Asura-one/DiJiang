# 变更影响矩阵

先识别项目现有的架构、接入、运维、决策和发布文档，再把对应受众更新到位。

## 先删的噪音

| 反模式 | 处理 |
|---|---|
| 纯历史叙事塞在规则文件顶部 | 现役约束就地融合；历史迁 git/changelog |
| 主规则里抄完整数据流 | 留边界和权威文档指针，机制回 docs |
| 多个版本都自称现役 | 以代码开关和运行态裁决；历史版标退役 |
| 已完成待办仍列开放项 | 核实后删除或改为当前约束 |
| 多处复制同一规则 | 保留一个权威，其他删除或放短指针 |
| Agent 会话残留文件 | 有效内容并入正式文档；文件列删除候选 |

## 代码变化 → 受影响知识面

| 变化 | 要核对的面 |
|---|---|
| API/路由/协议 | 外部接入合同、架构路由、鉴权、示例、smoke |
| 环境变量/provider/model | 示例 env、运行手册、成本/回退说明 |
| schema/存储字段 | 数据模型、迁移、缓存、读写边界、测试 |
| 用户流程/权限/导航 | README、架构权限、运营手册 |
| 评分/筛选/prompt | 现役算法文档、阈值、回测证据 |
| 服务/部署/CDN | 运维手册、服务名、marker、回滚 |
| 退役/改名/下线 | 搜旧 symbol 的非历史引用、consumer、rules、skills、memory |

## 规则违规 → 动作

| 发现 | 动作 |
|---|---|
| 规则要求同源但软链/导入缺失 | 补齐并验证加载 |
| 有代码但缺项目规则 | 创建最小规则文件（五要素、精简） |
| 命名违规、根目录乱放 | 只读报告影响；移动/重命名待决 |
| 同一违规反复出现 | 用 check/hook/test 根治 |

## 跨项目检查

共享 API、SDK、认证、schema 或部署改变时，搜索实际 consumer。只读发现不等于获准编辑。
