# 架构决策记录

## 目的

ADR 记录重要决策的原因。它不是设计文档，不应描述整个当前系统。DiJiang 使用 ADR 使决策历史持久化，同时允许设计文档、spec 和代码持续演进。

## 规则

- 对改变架构、工作流、公开契约、长期约定或默认 Agent 行为的决策编写 ADR。
- 一个 ADR 对应一个决策。如果一个主题需要多个独立选择，拆分为多个 ADR。
- ADR 编号永久稳定。不要重命名已接受的 ADR 来填补编号空缺。
- 明确追踪状态：`proposed`（提议）、`accepted`（已接受）、`rejected`（已拒绝）、`deprecated`（已弃用）或 `superseded`（已被取代）。
- 用取代而非重写历史。旧 ADR 指向新 ADR；新 ADR 注明它取代了哪个旧 ADR。
- 记录上下文、决策驱动因素、考虑过的备选方案、最终决策、后果和参考来源。
- 设计文档描述当前结构。它们可以引用 ADR 来给出原因，但不应重复 ADR 历史。
- 实现计划在决策约束工作时引用 ADR。

## 模板

```markdown
---
id: ADR-0001
title: 使用 <决策>
date: YYYY-MM-DD
status: proposed
supersedes: []
supersededBy: null
---

# ADR-0001: 使用 <决策>

## 上下文

什么问题或压力迫使现在做出决策？

## 决策驱动因素

- 因素 1
- 因素 2

## 考虑过的选项

| 选项 | 优点 | 缺点 | 结果 |
|--------|------|------|---------|
| 选项 A | ... | ... | accepted/rejected |

## 决策

选择的选项及其胜出原因。

## 后果

- 积极后果
- 负面或运维成本
- 对 spec、skill 或代码的后续约束

## 参考

- 相关 PRD、设计文档、Issue、Commit 或外部来源
```

## 示例

好的 ADR 主题：

- `ADR-0004: 使用 dj-check 作为规范质量门禁`
- `ADR-0008: 将任务状态存储在 .dijiang/tasks 而非旧版 .trellis`
- `ADR-0012: 要求任务 worktree 用于代码修改`

不好的 ADR 主题：

- `系统架构概述`
- `task crate 如何实现`
- `当前 CLI 命令列表`

## 反模式

- 编写大型设计文档并称之为 ADR。
- 编辑已接受的 ADR，直到旧引用不再能解释原始决策。
- 记录变更了什么，但不记录被拒绝的备选方案。
- 对属于代码或任务笔记的小型可逆实现细节使用 ADR。
