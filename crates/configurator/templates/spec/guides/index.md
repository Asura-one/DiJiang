# 思考指南

> **目的**：扩展你的思维，捕捉你可能没有考虑到的点。

---

## 为什么需要思考指南？

**大多数 bug 和技术债来自"没想到"**，而非技能不足：

- 没想到层边界会发生什么 → 跨层 bug
- 没想到代码模式会重复 → 到处都是重复代码
- 没想到边界情况 → 运行时错误
- 没想到未来的维护者 → 不可读的代码

这些指南帮助你在编码前**提出正确的问题**。

## 快速参考：思考触发器

### 何时思考跨层问题

- [ ] 功能涉及 3 个以上层（API、Service、Component、Database）
- [ ] 层之间数据格式发生变化
- [ ] 多个消费者需要相同的数据
- [ ] 不确定某个逻辑应该放在哪里
- [ ] 正在添加事件类型、JSONL 记录、RPC 载荷或配置字段
- [ ] UI / 命令代码开始直接转换原始载荷字段

→ 阅读[跨层思考指南](./cross-layer-thinking-guide.md)

### 何时思考代码复用

- [ ] 正在编写与已有代码相似的代码
- [ ] 同一个模式重复出现 3 次以上
- [ ] 正在向多个地方添加新字段
- [ ] **正在修改任何常量或配置**
- [ ] **正在创建新的工具/辅助函数** ← 先搜索！
- [ ] 两个文件使用局部转换读取相同的无类型载荷字段
- [ ] 多个分支从 `kind` / `action` 更新相同的派生状态

→ 阅读[代码复用思考指南](./code-reuse-thinking-guide.md)

### 验证 AI 交叉审查结果时

- [ ] 审查者声称"用户输入可能被恶意利用" → 检查实际数据来源（内部 manifest？用户配置？外部 API？）
- [ ] 审查者标记"缺少校验" → 数据是否来自受信任的内部来源？
- [ ] 审查者说"行为变更" → 阅读代码注释——是否是有意设计？
- [ ] 审查者识别出测试中的"bug" → 在脑海中删除被测试的功能——测试是否仍然通过？如果是 → 循环测试

**常见的 AI 审查误报模式**：
1. **信任边界混淆**：将内部数据（打包的 JSON manifests）当作不受信任的外部输入
2. **忽略设计注释**：将代码注释中记录的有意行为标记为 bug
3. **变量误读**：未追踪变量到其实际定义（例如，以路径而非名称作为键的 Map）

**验证规则**：CRITICAL/WARNING 级别的审查发现必须先对照实际代码验证，然后再确定优先级。AI 审查的误报率预算约为 35%。

### 构建验证循环时

- [ ] 任务有用户可见的或行为变更的目标
- [ ] 任务来自 AI 生成的代码或 AI 生成的 bug 修复
- [ ] 正在迁移旧行为、标签、字段或报告
- [ ] 目标环境包括浏览器、硬件、外部服务或生成的文件
- [ ] 当前的唯一证据是"代码看起来合理"

→ 阅读[验证循环指南](./verification-loop-guide.md)

### 编写持久化记忆时

- [ ] 即将编写 `dijiang mem findings` 或 `dijiang mem learn`
- [ ] 任务产出了可复用的策略、重复出现的失败或用户偏好
- [ ] 两个记忆、spec、文档或 task artifacts 之间存在冲突
- [ ] 希望未来的 agent 因为这个任务而有不同的行为
- [ ] 正在决定某内容属于记忆、spec、skill、ADR 还是任务笔记

→ 阅读[记忆生命周期指南](./memory-lifecycle-guide.md)

## 修改前规则（关键）

> **在更改任何值之前，务必先搜索！**

```bash
# 搜索你要更改的值
grep -r "要更改的值" .
```

这一个习惯可以防止大多数"忘记更新 X"的 bug。

## 如何使用本目录

1. **编码前**：浏览相关思考指南
2. **编码中**：如果感觉代码重复或复杂，查阅指南
3. **出 bug 后**：将新见解添加到相关指南（从错误中学习）
