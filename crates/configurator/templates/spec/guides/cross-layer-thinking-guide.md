# 跨层思考指南

> **目的**：实现前思考跨层数据流。

---

## 问题

**大多数 bug 发生在层边界**，而非层内部。

常见的跨层 bug：

- API 返回格式 A，前端预期格式 B
- 数据库存储 X，服务转换为 Y，但数据丢失
- 多个层以不同方式实现相同逻辑

---

## 实现跨层功能前

### 第一步：绘制数据流

画出数据的移动路径：

```
来源 → 转换 → 存储 → 检索 → 转换 → 展示
```

对每个箭头，问：

- 数据是什么格式？
- 可能会出什么问题？
- 谁负责校验？

### 第二步：识别边界

| 边界 | 常见问题 |
| --------------------- | --------------------------------- |
| API ↔ Service | 类型不匹配、字段缺失 |
| Service ↔ Database | 格式转换、null 处理 |
| Backend ↔ Frontend | 序列化、日期格式 |
| Component ↔ Component | Props 形状变更 |

### 第三步：定义契约

对每个边界：

- 确切的输入格式是什么？
- 确切的输出格式是什么？
- 可能发生哪些错误？

---

## 常见跨层错误

### 错误 1：隐式格式假设

**糟糕的**：假设日期格式而不检查

**好的**：在边界处显式转换格式

### 错误 2：校验分散

**糟糕的**：在多个层校验同一件事

**好的**：在入口点校验一次

### 错误 3：抽象泄漏

**糟糕的**：组件知道数据库 schema

**好的**：每层只认识它的相邻层

### 错误 4：每个消费者都在解析相同的载荷

**糟糕的**：命令读取 JSONL 事件并内联转换字段：

```typescript
const thread = (ev as { thread?: string }).thread;
const labels = (ev as { labels?: string[] }).labels;
```

这看起来是局部问题，但意味着每个消费者拥有事件契约的私有版本。下次字段变更会更新一个命令而遗漏另一个。

**好的**：在事件边界解码一次，然后导出带类型的投影：

```typescript
if (!isThreadEvent(ev)) return false;
return ev.thread === filter.thread;
```

**规则**：对于追加日志、JSON 流、RPC 载荷或配置文件，为以下内容创建单一所有者：

- 事件/载荷类型定义
- 类型守卫和来自 `unknown` 的规范化
- UI 命令使用的元数据投影
- 从事实验证源重放状态的 reducer

渲染代码可以格式化字段，但不能重新定义载荷契约。

## 跨层功能清单

实现前：

- [ ] 绘制了完整数据流
- [ ] 识别了所有层边界
- [ ] 定义了每个边界的格式
- [ ] 决定了校验发生的位置

实现后：

- [ ] 用边界情况测试（null、空、无效）
- [ ] 验证了每个边界的错误处理
- [ ] 检查数据是否完整往返
- [ ] 检查消费者是否导入共享解码器/投影，而非本地转换载荷字段
- [ ] 检查派生状态是否指向源事件标识符（`seq`、`id`、`version`），而非发明第二个游标

## 跨平台模板一致性

在 Trellis 中，命令模板（如 `record-session.md`）存在于**多个平台**中，内容相同或几乎相同。这是一个跨层边界。

### 修改命令模板后的清单

- [ ] 查找所有包含相同命令的平台：`find src/templates/*/commands/trellis/ -name "<命令>.*"`
- [ ] 更新所有平台副本（Markdown `.md` 和 TOML `.toml`）
- [ ] 对于 Gemini TOML：调整行续接（`\\` 与 `\`）和三引号字符串
- [ ] 运行 `/trellis:check-cross-layer` 确认没有遗漏

**真实案例**：在 Claude 中更新了 `record-session.md` 使用 `--mode record`，但忘记了 iFlow、Kilo、OpenCode 和 Gemini——被跨层检查发现。

---

## 生成的运行时模板升级一致性

一些生成的文件既是文档也是运行时输入。在 Trellis 中，`.trellis/workflow.md` 被 `get_context.py`、`workflow_phase.py`、SessionStart 过滤器和每轮 hook 解析。模板变更必须同时在全新 init 和升级路径中验证。

### 修改运行时解析模板后的清单

- [ ] 识别读取该模板的每个运行时解析器，而非仅仅安装它的文件写入器
- [ ] 检查相关语法是否位于明显管理区域之外（如标签块）
- [ ] 验证全新 `init` 输出和版本化 `update` 场景（写入旧版 `.trellis/.version`）
- [ ] 使用旧的原始模板 fixture 添加升级回归测试，断言安装后的文件达到当前打包形态
- [ ] 更新拥有运行时契约的后端 spec

## 版本化文档边界

版本化文档是一个跨层边界：源路径、`docs.json` 版本路由和渲染后的版本选择器必须描述相同的发布线。

### 编辑版本化文档前的清单

- [ ] 确定目标发布线：stable、beta 或 RC
- [ ] 验证编辑的 MDX 路径与该发布线匹配：
  - stable：`docs-site/{start,advanced,...}` 和 `docs-site/zh/{start,advanced,...}`
  - beta：`docs-site/beta/**` 和 `docs-site/zh/beta/**`
  - RC：`docs-site/rc/**` 和 `docs-site/zh/rc/**`
- [ ] 验证 `docs.json` 导航将版本标签指向相同路径
- [ ] 提交前对相对树的发布线特定术语进行 grep
- [ ] 将根发布路径下的 beta 内容视为源路径 bug，而非渲染 bug

**真实案例**：仅针对 beta 的任务工作流变更将 `prd.md` + `design.md` + `implement.md`、任务创建确认和 Codex 模式横幅记录在根路径 `start/` 和 `advanced/` 下。文档站点随后在 Release 选择器下提供了 0.6 beta 行为。修复方法：恢复根发布文档，将 0.6 内容移至 `beta/` 和 `zh/beta/`，并添加针对根发布树的 beta 标记 grep 审计。

**真实案例**：Codex 内联模式将工作流平台标记从 `[Codex]` / `[Kilo, Antigravity, Windsurf]` 改为 `[codex-sub-agent]` / `[codex-inline, Kilo, Antigravity, Windsurf]`。全新 init 正确，但 `trellis update` 只合并了 `[workflow-state:*]` 块，保留了这些块之外的过期标记。结果：升级后的项目获得了新的 hook 脚本但保留了旧的工作流路由，导致 `get_context.py --mode phase --platform codex` 可能返回空的 Phase 2.1 详情。

## 模式检测探测清单

当 CLI 通过探测远程资源自动检测模式时（例如，检查 `index.json` 是否存在以决定 marketplace 与直接下载）：

### 实现前：

- [ ] 探测在**所有**使用该结果的代码路径中运行（交互式、`-y`、`--flag` 组合）
- [ ] 区分 404 和临时错误——不要将两者都视为"未找到"
- [ ] 临时错误应**中止或重试**，永远不要静默切换模式
- [ ] 共享状态（缓存、预取数据）在上下文变化时（如用户切换源）被**重置**
- [ ] **快捷路径**（如 `--template` 跳过选择器）必须具有与探测路径相同的错误处理质量——检查下游函数是否没有调用全捕获包装器

### 实现后：

- [ ] 追踪从探测结果到模式决策分支的每条路径——不允许 fallthrough
- [ ] 外部格式契约（giget URI、原始 URL）需经过测试或至少在注释中记录
- [ ] 元数据读取消耗完整响应或使用流式解析器——不要将固定大小的前缀解析为完整 JSON
- [ ] 当从解析部分重构复合标识符时，验证**所有**字段都包含且**位置正确**（例如 `provider:repo/path#ref` 而非 `provider:repo#ref/path`）
- [ ] 验证快捷方式后调用的**动作函数**不会内部使用旧的全捕获 fetch——当错误区分重要时，它们必须使用探测质量的变体

**真实案例**：自定义注册表流程在 3 轮审查中发现 8 个 bug：(1) 探测仅在交互模式下运行，(2) 临时错误落到错误模式，(3) giget URI 中 `#ref` 位置错误，(4) 预取模板跨源泄漏，(5) `--template` 快捷方式绕过了探测，但 `downloadTemplateById` 内部使用全捕获 `fetchTemplateIndex`，将超时转为"未找到模板"。

**真实案例**：Agent-session 更新提示通过 `response.read(4096)` 获取 npm `latest` 元数据，然后将其解析为完整 JSON。`@mindfoldhq/trellis` 包元数据超过 4 KB，JSON 被截断，解析静默失败，首次 session 注入未显示更新提示。修复方法：解析前读取完整响应，并添加一个回归测试，其中 `version` 后跟 8 KB 元数据尾。

## 何时创建流程文档

在以下情况创建详细的流程文档：

- 功能涉及 3 个以上层
- 涉及多个团队
- 数据格式复杂
- 该功能之前曾引发 bug

## 事件日志/投影边界

追加日志是跨层契约。单个事件经历了以下路径：

```
CLI 输入 → 事件写入器 → events.jsonl → 读取器 → 过滤器 → 归约器 → 展示
```

### 添加新事件类型或字段后的清单

- [ ] 将事件类型添加到中央事件分类法
- [ ] 在事件层添加带类型的事件变体或类型守卫
- [ ] 为来自用户输入或 JSON 的数组/对象字段添加规范化辅助函数
- [ ] 仅在事件写入器中维护 `seq` / `id` 赋值
- [ ] 让过滤器和归约器消费带类型的事件守卫，而非本地转换
- [ ] 让展示代码消费归约器输出或带类型的事件，而非原始 JSON
- [ ] 添加至少一个回归测试，证明历史重放和实时过滤使用相同的过滤器模型

**真实案例**：线程频道添加了 `kind: "thread"`、`description`、`context`、labels 和 `lastSeq`。首次实现正确重放了线程状态，但几个命令仍使用本地转换重新解析事件载荷字段。修复方法：让核心事件层拥有 `ThreadChannelEvent` 和 `isThreadEvent`，让 `reduceChannelMetadata` 成为唯一的频道元数据投影，让 `reduceThreads` 成为唯一的线程重放归约器。
