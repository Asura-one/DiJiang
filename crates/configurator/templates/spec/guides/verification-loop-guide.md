# 验证循环指南

## 目的

验证循环将目标转化为可观察的通过/失败信号。DiJiang 的工作应通过证据而非信心来收敛。当任务模糊、AI 生成的代码需要验证或变更涉及用户可见的行为时，使用此指南。

## 规则

- 在编码前定义最小的关键命题：任务完成时什么必须为真？
- 选择能够证明该命题的、最便宜的可靠循环：单元测试、CLI fixture、HTTP 调用、浏览器/OCR 检查、trace 重放或人工检查清单。
- 验证必须能在单条命令或一个可重复的步骤中执行。
- 如果开始时无法确定验证方法，则设计可以观察运行间差异的实验：用最小变更对比前后行为。
- 修复 bug 时：先编写能复现失败的验证（RED），然后再修复（GREEN）。
- AI 生成的变更必须至少通过机器可执行的验证，才能声称"合理"。
- 注释 `not run` + 明确原因，而不是假装检查覆盖了它。
- 当使用人工检查清单时：每个检查项必须是可观察的（"表单提交后出现成功消息"），而非主观的（"看起来不错"）。

## 常见错误

- 选择验证但从未实际运行它 → 它不是验证，是意图声明。
- 只验证快乐路径——边界情况和错误路径需要独立的循环。
- 对简单的结构化转换（重命名、lint 修复、机械重构）进行过度验证——一次编译确认通常就足够了。
- 将"不需要验证"作为跳过思考的借口——在依赖推理之前，先做一个便宜的可观察检查。

## 示例

| 任务 | 验证 | 反馈循环 |
|------|------------|---------------|
| 添加 API 端点 | `curl` 返回预期 JSON | HTTP 返回码 + 响应体 |
| 修改数据库 schema | `dijiang migrate test` 返回 0 | 迁移工具测试 |
| 修复 UI 状态 bug | 点击 → 状态显示预期值 | 浏览器检查 + 截图对比 |
| 重构函数 | 现有测试全部通过 | `make test` |
| AI 生成的组件 | 渲染并通过视觉检查 | `npm run build` + 截图 diff |
