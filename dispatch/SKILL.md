# Dispatch: 任务分流器

## 职责

收到用户请求后，判断任务级别，走对应路径。不纠结、不拖沓。

## 判断标准

### S级（零碎）— 直接干

特征：
- 一句话能说清（"改个变量名"、"加个日志"、"这个 typo 修一下"）
- 影响范围 < 3 个文件
- 不涉及架构、不需要调研、不需要设计

具体判断：
- 修改单个文件的单个位置（变量名、日志、注释、typo）
- 修改 1-2 个文件的简单逻辑（加个条件判断、改个返回值）
- 不涉及新功能、不涉及跨模块依赖

路径：
```
直接执行 → 完成
```

### M级（中等）— 快速确认后干

特征：
- 范围明确但需要一些思考（"加个字段校验"、"修这个样式"、"重构这个函数"）
- 影响 3-10 个文件
- 不涉及新模块/新架构，但需要确认方向

具体判断：
- 修改 3-10 个文件的中等复杂度任务
- 涉及现有模块的局部重构
- 需要确认技术方案（如：用哪个库、怎么处理边界情况）

路径：
```
grill（3个问题以内快速确认）→ 实现 → 完成
```

### L级（完整）— 走完整流程

特征：
- 新功能、架构改动、不确定的需求
- 影响 > 10 个文件或跨模块
- 需要调研、设计、PRD
- 用户自己也不完全确定要什么

具体判断：
- 新增模块/功能（如：加个支付模块、加个搜索功能）
- 跨模块改动（如：重构整个认证流程）
- 需求不明确，需要深度调研
- 涉及架构决策（如：选数据库、选消息队列）

路径：
```
grill（深度调研+细化）→ output（PRD+设计文档）→ implement/dj-tdd → hunt ↔ check（循环）→ 完成
```

## 判断流程

**Think Before Coding**：先理解用户意图，不要急于执行。问自己："用户真正想要什么？"

1. 读用户请求
2. 按上述标准判断级别
3. 告知用户判断结果：
   ```
   任务级别：[S/M/L]
   推荐路径：[直接干 / grill快速确认 / 完整流程]
   ```
4. **S级**：直接执行，不废话
5. **M级**：调用 `grill`，限定 3 个问题以内
6. **L级**：调用 `grill`，不限问题数

## 边界

- 判断不了时，偏保守选高一级（M→L，S→M）
- 用户明确指定流程时（"走完整流程"、"直接干"），跳过判断按用户说的来
- 如果用户只是问问题/聊天，不触发 dispatch

## 失败处理

| 触发条件 | 一线修复 | 仍失败兜底 |
|---------|---------|-----------|
| 判断不了是 S/M/L | 偏保守选高一级（S→M, M→L） | 问用户"这个任务你想要什么粒度？" |
| 用户给了模糊任务 | 按任务描述的最长路径估计 | 调用 grill 快速确认 3 个问题 |
| grill 超时/不可用 | 跳过 grill，直接按现有信息 implement | 告知用户"grill 不可用，按现有理解执行" |
| 用户指定的流程和判断冲突 | 按用户说的来 | 告知判断差异，但执行用户指令 |

## 🔴 CHECKPOINT · 分流确认

M/L 级任务分流后：
```
任务级别：<S/M/L>
推荐路径：<具体流程>
预估时间：<分钟>

确认？(Y/n)
```

## 反例

| ❌ 不要做 | ✅ 正确做法 |
|---|---|
| S级任务走完整流程（过度工程） | 一句话能解决的事直接干 |
| L级任务跳过 grill 直接写代码 | 不确定的需求必须先对齐 |
| 判断为 M 但不告知用户 | 告知判断结果和推荐路径 |
| 用户说"直接干"还要走流程 | 尊重用户意图，跳过判断 |
