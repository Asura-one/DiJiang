# Dispatch: 任务分流器

## 职责

收到用户请求后，判断任务级别，走对应路径。不纠结、不拖沓。

## 判断标准

### S级（零碎）— 直接干

特征：
- 一句话能说清（"改个变量名"、"加个日志"、"这个 typo 修一下"）
- 影响范围 < 3 个文件
- 不涉及架构、不需要调研、不需要设计

具体判断：
- 修改单个文件的单个位置（变量名、日志、注释、typo）
- 修改 1-2 个文件的简单逻辑（加个条件判断、改个返回值）
- 不涉及新功能、不涉及跨模块依赖

路径：
```
直接执行 → 完成
```

### M级（中等）— 快速确认后干

特征：
- 范围明确但需要一些思考（"加个字段校验"、"修这个样式"、"重构这个函数"）
- 影响 3-10 个文件
- 不涉及新模块/新架构，但需要确认方向

具体判断：
- 修改 3-10 个文件的中等复杂度任务
- 涉及现有模块的局部重构
- 需要确认技术方案（如：用哪个库、怎么处理边界情况）

M 级内部再分两种：

**M-simple（简单 M）**：
- 一次性改完就能交付
- 改动之间没有依赖，不需要逐步验证
- 只涉及单一层面（只改数据/只改接口/只改前端/只改配置）
- 例：加个字段校验、改个样式、重构一个函数、加个环境变量

**M-phased（分阶段 M）**：
- 需要分 2-3 步实现，每步需要验证
- 步骤之间有依赖，前一步的产出是后一步的输入
- 涉及 2 层以上（数据层+接口层+展示层，或旧代码迁移+新功能）
- 例：先搭数据模型 → 再加 API → 再接前端
- 例：先迁移旧代码 → 再加新功能 → 再清理旧路径

**快速判断**：改动只涉及 1 层 → M-simple；涉及 2+ 层 → M-phased。

路径：
```
M-simple:  直接实现 → 完成
M-phased:  grill（≤3问，一次确认）→ dj-tdd（红绿重构）→ 完成
```

**用户可覆盖**：用户说"走 TDD"→ 按 M-phased 走；用户说"不走 TDD"→ 按 M-simple 走。

### L级（完整）— 走完整流程

特征：
- 新功能、架构改动、不确定的需求
- 影响 > 10 个文件或跨模块
- 需要调研、设计、PRD
- 用户自己也不完全确定要什么

具体判断：
- 新增模块/功能（如：加个支付模块、加个搜索功能）
- 跨模块改动（如：重构整个认证流程）
- 需求不明确，需要深度调研
- 涉及架构决策（如：选数据库、选消息队列）

路径：
```
grill（深度对齐）→ 🔴确认1 → output（PRD+设计文档）→ 🔴确认2 → dj-tdd（红绿重构）→ hunt ↔ check（循环）→ 完成
```

## 判断流程

**Think Before Coding**：先理解用户意图，不要急于执行。问自己："用户真正想要什么？"

1. 读用户请求
2. 按上述标准判断级别
3. 告知用户判断结果：
   ```
   任务级别：[S / M-simple / M-phased / L]
   推荐路径：<具体流程>
   ```
4. **S级**：直接执行，不废话
5. **M-simple**：直接实现，不走 grill
6. **M-phased**：调用 `grill`，限定 3 个问题以内
7. **L级**：调用 `grill`，不限问题数

## 流水线衔接（关键）

**核心原则**：做完事再汇报。不征求继续许可，不问"要不要我做下一步"。

### S 级
- 直接干，完成后告知用户

### M-simple 流水线
```
直接实现 → 完成
```
- 不走 grill，不写 PRD，不走 TDD
- 告知用户"任务级别：M-simple，直接实现"，然后开干

### M-phased 流水线
```
grill（≤3问）→ 🔴 一次确认 → dj-tdd（红绿重构）→ 完成
```
- grill 完成后输出需求摘要，**只确认一次**
- 用户确认后，自动进入 dj-tdd，不再停顿
- TDD 完成后直接告知结果，不问"要不要审查"

### L 级流水线
```
grill（深度对齐）→ 🔴 确认1 → output（PRD）→ 🔴 确认2 → dj-tdd（红绿重构）→ check → 完成
```
- grill 完成后输出需求摘要，等用户确认
- 用户确认后，加载 `output` skill 写 PRD+设计文档
- PRD 完成后等用户确认，然后进入 dj-tdd 实现
- **L 级强制 TDD**：必须先写测试再写实现，红绿重构循环，不允许跳过
- TDD 完成后自动加载 `check` skill 做代码审查（不停顿）
- 发现问题自动加载 `hunt` skill 排查根因

### 🔴 检查点

检查点数量按级别递增：

| 级别 | 检查点 | 位置 |
|------|--------|------|
| S 级 | 0 | 无 |
| M-simple | 0 | 无 |
| M-phased | 1 | grill → dj-tdd |
| L 级 | 2 | grill → output、output → dj-tdd |

检查点内容：
```
需求摘要：
- 目标：<一句话>
- 范围：<包含/不包含>
- 关键决策：<技术选型>

确认后按此执行？(Y/n)
```

- dj-tdd → check 不停顿，自动推进
- 用户说"n"：停下来讨论，不强行继续
- 用户说"跳过"：用推荐答案填充未决项，继续推进

### 全速模式

用户说"直接干"、"不用问了"、"你决定"等意图明确的指令时：
- 跳过所有检查点（包括 grill 的需求确认）
- 按最短路径执行：S→直接干，M→直接实现，L→grill 用推荐答案填充 → output → dj-tdd → check
- 完成后告知结果

## 边界

- 判断不了时，偏保守选高一级（M→L，S→M）
- 用户明确指定流程时（"走完整流程"、"直接干"），跳过判断按用户说的来
- 如果用户只是问问题/聊天，不触发 dispatch
- **唯一正当的暂停理由**：存在真正的歧义，继续工作会产出违背用户意图的结果

## 失败处理

| 触发条件 | 一线修复 | 仍失败兜底 |
|---------|---------|-----------|
| 判断不了是 S/M/L | 偏保守选高一级（S→M, M→L） | 问用户"这个任务你想要什么粒度？" |
| 分不清 M-simple 还是 M-phased | 按 M-phased 走（偏保守） | 问用户"需要分步验证吗？" |
| 用户给了模糊任务 | 按任务描述的最长路径估计 | 调用 grill 快速确认 3 个问题 |
| grill 超时/不可用 | 跳过 grill，直接按现有信息 implement | 告知用户"grill 不可用，按现有理解执行" |
| 用户指定的流程和判断冲突 | 按用户说的来 | 告知判断差异，但执行用户指令 |
| 用户中途改变需求 | 重新判断级别（可能升/降级） | 告知用户"需求变了，重新评估级别" |
| TDD 循环卡住（测试连续3轮不过） | 暂停 TDD，诊断根因 | 加载 `hunt` skill 排查，或降级为直接实现 |
| 用户反复切换任务级别 | 锁定当前判断，展示判断依据 | 问用户"你想要什么级别？我按你说的来" |

## 反例

| ❌ 不要做 | ✅ 正确做法 |
|---|---|
| S级任务走完整流程（过度工程） | 一句话能解决的事直接干 |
| L级任务跳过 grill 直接写代码 | 不确定的需求必须先对齐 |
| 判断为 M 但不告知用户 | 告知判断结果和推荐路径 |
| 用户说"直接干"还要走流程 | 尊重用户意图，跳过所有检查点 |
| grill 完成后问"要不要继续" | 确认后自动推进，不征求继续许可 |
| M-simple 走 grill 浪费时间 | M-simple 直接实现，不过度对齐 |
| 每步转场都问"确认？" | 只在需求确认点停顿，其余自动推进 |
| 用户说"你决定"还在追问 | 用推荐答案填充，继续推进 |
