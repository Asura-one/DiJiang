# Skill Optimization Design

## 技术方案

使用 darwin-skill 的自主优化循环对项目 skill 进行评审和优化。

## 评估框架

### 9 维度评估 rubric

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 1. Frontmatter 质量 | 7 | name 规范、description 包含做什么+何时用+触发词、≤1024 字符 |
| 2. 工作流清晰度 | 12 | 步骤明确可执行、有序号、每步有明确输入/输出 |
| 3. 失败模式编码 | 12 | 必须显式编码失败模式、有 fallback 路径、错误恢复 |
| 4. 检查点设计 | 6 | 关键决策前有用户确认、防止自主失控 |
| 5. 可执行具体性 | 17 | 不模糊、有具体参数/格式/示例、可直接执行 |
| 6. 资源整合度 | 4 | references/scripts/assets 引用正确、路径可达 |
| 7. 整体架构 | 12 | 结构层次清晰、不冗余不遗漏 |
| 8. 实测表现 | 23 | 用测试 prompt 跑一遍，输出质量是否符合 skill 宣称的能力 |
| 9. 反例与黑名单 | 6 | skill 必须有"不要做什么"的反例清单 |

### 评估流程

1. **Phase 0: 初始化**
   - 确认优化范围（全部或部分 skill）
   - 创建 git 分支：auto-optimize/YYYYMMDD-HHMM
   - 初始化 results.tsv（如果不存在）

2. **Phase 0.5: 测试 Prompt 设计**
   - 为每个 skill 设计 2-3 个典型用户 prompt
   - 保存到 skill 目录/test-prompts.json
   - 展示给用户确认

3. **Phase 1: 基线评估**
   - 结构评分（维度 1-7、9）：主 agent 评估
   - 效果评分（维度 8）：独立子 agent 评估
   - 计算加权总分并记录到 results.tsv

4. **Phase 2: 优化循环**
   - 按分数从低到高排序，优先优化最低 skill
   - 每轮只改一个维度
   - 重新评估，只保留改进（棘轮机制）
   - 每个 skill 优化完后强制人审

5. **Phase 3: 汇总报告**
   - 生成优化报告和成果卡片
   - 记录到 results.tsv

## 优化策略

按优先级排序：
- P0: Runtime 适配性问题（gate 项命中 → 必须先修）
- P0: 效果问题（实测发现的）
- P1: 结构性问题
- P2: 具体性问题
- P3: 可读性问题

## 约束规则

1. 不改变 skill 的核心功能和用途
2. 不引入新依赖
3. 每轮只改一个维度
4. 保持文件大小合理（优化后不超过原始大小的 150%）
5. 尊重花叔风格（中文为主、简洁为上）
6. 可回滚（所有改动在 git 分支上）
7. 评分独立性（效果维度必须用独立子 agent 评估）
8. Runtime 中立性（skill 必须能在任何 skills-compatible runtime 中正常运行）
