# Karpathy 编码准则 Skill 有效性评估报告

## 评估概要

针对「为 Python API 调用函数添加错误处理」这一典型任务，Karpathy 编码准则 skill 展现出显著的引导效果。Skill-Guided 方案在简洁性和可读性上获得满分（10/10），总分 46 分，较 Baseline 的 39 分提升 18%。核心差异在于错误处理策略的选择：Baseline 倾向于区分异常类型并建议自定义异常类，符合「全面覆盖」的工程直觉；Skill-Guided 则遵循「简单优先」原则，统一返回 None，避免了过度工程。这一对比验证了 skill 在规避 LLM 常见反模式（过度抽象、样板代码膨胀）方面的有效性。

## Baseline vs Skill-Guided 对比

| 维度 | Baseline | Skill-Guided | 差异说明 |
|------|----------|--------------|----------|
| 意图完成度 | 9 | 8 | Baseline 覆盖更多边缘场景，Skill-Guided 略有取舍 |
| 简洁性 | 6 | 10 | Skill-Guided 代码量显著减少，无冗余结构 |
| 实用性 | 9 | 9 | 两者均能直接用于生产，实用性持平 |
| 可读性 | 7 | 10 | Skill-Guided 单次阅读即可理解，认知负担低 |
| 反模式规避 | 8 | 9 | Skill-Guided 明确规避自定义异常、过度区分等反模式 |
| **总分** | **39** | **46** | **Skill-Guided 综合表现更优** |

**错误处理策略对比：**

- **Baseline**：区分 `ConnectionError`、`Timeout`、`HTTPError` 等异常类型，建议引入自定义异常类包装底层错误
- **Skill-Guided**：统一捕获异常并返回 None，调用方按需处理，符合 YAGNI 原则

## Skill 有效性分析

**有效性评分：7/10**

Skill 在本次测试中发挥了明确的正向引导作用：

1. **简洁性提升显著**：从 6 分跃升至 10 分，说明 skill 对「一次阅读能理解」这一原则的传导效果强
2. **反模式规避有效**：skill 明确列举的「到处 try/catch」「自定义异常滥用」等反模式在 Skill-Guided 方案中均未出现
3. **YAGNI 执行到位**：不过早抽象的原则被准确贯彻，未出现为假想场景预留的扩展点

**局限性**：Skill-Guided 在意图完成度上略低于 Baseline（8 vs 9），说明 skill 的简化倾向可能在某些场景下牺牲了必要的覆盖度。

## 潜在负面影响

当前任务为简单 API 调用函数，Skill-Guided 的简化策略完全适用。但存在以下风险：

1. **复杂场景适应性不足**：若调用方需要区分「网络不可用」与「认证失败」以执行不同恢复策略，统一返回 None 的方案将无法满足需求
2. **错误信息丢失**：吞掉异常而不记录，可能增加生产环境的调试难度
3. **skill 边界模糊**：skill 未提供「何时应增加复杂度」的判断标准，使用者可能在本应精细化处理的场景中过度简化

本次测试未观察到实际负面影响，但上述风险在更复杂的工程场景中可能显现。

## 改进建议

1. **增加复杂度梯度指引**：在 skill 中补充「简单 → 中等 → 复杂」场景的判断标准，例如：
   - 简单：单一调用方、无恢复逻辑 → 统一返回 None
   - 中等：多调用方、需日志记录 → 捕获 + 日志 + 返回 None
   - 复杂：需差异化恢复 → 区分异常类型，但不自定义异常类

2. **强化日志建议**：在「统一返回 None」策略中默认建议添加 `logging.warning` 或 `logging.error`，避免错误信息完全丢失

3. **补充反例**：在 skill 中加入「过度简化也是一种反模式」的说明，防止矫枉过正

|------|------|
| Skill 引导有效性 | 7/10 |
| 简洁性改善幅度 | 4/4（6→10） |
| 反模式规避效果 | 1/1（8→9） |
| 场景适应性 | 3/5（简单场景优秀，复杂场景存疑） |

**结论**：Karpathy 编码准则 skill 在简单到中等复杂度的任务中表现出色，能有效抑制 LLM 倾向的过度工程行为。建议在 skill 中增加复杂度梯度指引，以提升其在多样化场景中的适用性。
